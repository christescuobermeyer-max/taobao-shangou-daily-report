# 淘宝闪购阶段二云端日报系统开发文档

**版本：** v1.1

**文档状态：** 已部署上线，并完成登录持久化、真实门店查询、日报生成和六运营并发验收。

**目标：** 将阶段一的淘宝闪购昨日数据抓取和日报生成能力部署到云服务器，运营输入门店 ID 后即可获取门店信息、昨日数据和大模型生成的店铺日报。

## 1. 方案确认

本阶段采用“云端常驻 Chrome + Node API + 静态运营页面”的方案。服务端负责登录淘宝闪购服务商后台、查询门店数据和调用大模型；浏览器页面只负责输入门店 ID、展示结果和复制文本。

本方案的关键验收要求是：**Cookie 必须实际保存在云服务器，云端 Chrome 必须能够用该 Cookie 登录淘宝闪购服务商页面，并且必须使用真实门店 ID 读取到昨日数据。** 仅上传 Cookie 文件、页面能打开、或接口返回固定示例数据，均不能算验收通过。

淘宝闪购后台当前使用 `open.shop.ele.me`、`.ele.me` 等历史技术域名，及 `ELEME_ISV_COOKIE` 历史变量名。这些是淘宝闪购服务商后台的实际技术标识，业务名称和页面展示统一使用“淘宝闪购”，不得在用户页面或日报中显示为饿了么。

本方案暂不增加应用登录口令，按公司内部使用场景部署。公网地址仍必须通过 HTTPS 访问；后续如访问范围扩大，应增加访问口令或网络白名单。

## 3. 云端敏感配置

云服务器应用目录内单独保存权限为 `600` 的 `.env`，仅 Node 服务进程读取：

```env
# 淘宝闪购服务商登录态，部署时从本地安全配置同步，不提交 Git
ELEME_ISV_COOKIE=...

# 日报大模型配置
VECTORENGINE_API_KEY=...
VECTORENGINE_BASE_URL=https://api.vectorengine.ai
VECTORENGINE_MODEL=gpt-5.6-luna

# 本地 Chrome/CDP
CHROME_DEBUG_PORT=9222
APP_PORT=8792
REPORT_BROWSER_TIMEOUT_MS=30000
REPORT_LLM_CONCURRENCY=3
```

Cookie 和 API Key 不发送到浏览器、不写入 API 响应、不写入日志。`.env.example` 只能包含变量名和占位值。

Cookie 的云端保存要求：

1. 实际值只写入云端应用 `.env` 的 `ELEME_ISV_COOKIE`，或权限为 `600` 的浏览器 Cookie JSON；JSON 路径由 `COOKIE_BUNDLE_PATH` 配置，不能写入 Git、开发文档、前端静态资源或普通运行日志。
2. Chrome 使用固定的持久化用户目录，例如 `/home/devbox/chrome-profile`，不能每次启动都创建临时 profile，否则 Cookie 无法跨重启复用。
3. 服务启动时从云端配置恢复 Cookie；服务重启后必须重新执行登录态检查，不能只依赖上一次内存状态。
4. Cookie 失效时不能静默使用旧缓存或示例数据，必须让健康检查和日报接口明确失败，并记录不包含敏感值的错误原因。

## 4. Cookie 登录闭环

Cookie 登录是阶段二的硬性前置条件，不能只把 Cookie 上传后就认为部署成功，必须完成以下闭环：

1. 部署脚本将 `ELEME_ISV_COOKIE` 写入云端应用 `.env`，设置文件权限 `600`。
2. 云端启动有头 Chrome，使用独立用户目录，例如 `/home/devbox/chrome-profile`，调试端口只监听 `127.0.0.1:9222`。
3. Node 服务从 `ELEME_ISV_COOKIE` 解析 Cookie，通过 CDP `Network.setCookies` 注入淘宝闪购服务商后台。
4. 服务打开 `https://open.shop.ele.me/manager/base/store-analysis`。
5. 服务检查页面是否存在“搜索门店名称/ID”输入框，以确认登录态可用。
6. 服务检查实际页面 URL、Cookie 注入结果和门店分析页面加载状态。
7. `GET /api/health` 返回：

```json
{
  "ok": true,
  "loggedIn": true,
  "chrome": true,
  "storeAnalysisPage": true
}
```

只有同时满足 `chrome: true`、`loggedIn: true`、`storeAnalysisPage: true`，才允许返回 `loggedIn: true`。健康检查至少要验证：

- 当前页面没有跳转到登录页或登录失效页；
- 页面存在“搜索门店名称/ID”输入框；
- 通过一次真实的服务商门店列表请求，确认登录态可以访问业务接口；
- 验证结果不包含 Cookie、token、API Key 或完整请求头。

Cookie 过期或风控失效时，健康检查必须返回 `loggedIn: false` 和可读错误，不允许继续调用大模型生成错误日报。

## 6. 服务端 API

### `GET /api/health`

检查 Chrome、Cookie 登录态和服务商门店分析页面。

### `GET /api/shop/:shopId`

返回门店基础信息：

```json
{
  "shopId": "1330475849",
  "shopName": "老碰私厨馆(汇川店)",
  "operator": "王清月",
  "groupName": "B1饿了么26/7.15老碰私厨馆(汇川店)8%"
}
```

### `POST /api/report`

请求：

```json
{ "shopId": "1330475849", "date": "20260716" }
```

处理流程：

1. 校验门店 ID 格式并读取门店基础信息。
2. 按 `shopId-date` 合并同一时刻的重复请求，并再次检查成功缓存。
3. 将取数任务放入单例 Chrome 队列，保证共享页面搜索串行。
4. 搜索门店 ID，只接受请求体 `searchWord` 与当前门店 ID 一致的 `pageQueryOpenOfflineDetails` POST 响应。
5. 保存完整原始数据，并将字段转换为中文数据名称。
6. 释放 Chrome 队列后，将模型任务放入默认并发 3 的独立队列。
7. 大模型失败或超时后重试 2 次，仍失败才返回错误。
8. 成功结果原子写入缓存并返回：

```json
{
  "shopId": "1330475849",
  "shopName": "老碰私厨馆(汇川店)",
  "operator": "王清月",
  "groupName": "B1饿了么26/7.15老碰私厨馆(汇川店)8%",
  "date": "20260716",
  "yesterdayData": "昨日日期：20260716\n...",
  "report": "淘宝闪购运营日报..."
}
```

`POST /api/report` 的成功条件不是“接口返回 200”，而是同时满足以下条件：

1. 云端服务已通过 Cookie 登录态检查；
2. 返回数据中的门店 ID 与请求门店 ID 完全一致；
3. 返回数据中的店铺名与云端 `shops.json` 和淘宝闪购页面结果一致；
4. 昨日日期为服务端计算的昨天，不能由前端任意伪造为未来日期；
5. 原始接口字段完整保存到 `yesterdayData`，并完成中文业务名称转换；
6. 日报确实由配置的大模型 API 生成，失败重试后仍失败则返回错误，不返回伪造日报。

服务端可以缓存同一门店同一天的成功结果，避免运营重复点击造成重复 API 消费；缓存文件不保存 Cookie 和 API Key。

### 6.1 六运营并发策略

- 浏览器/CDP 抓取固定并发 `1`：共享页面在任何时刻只查询一个门店，避免页面状态和接口响应串店。
- 大模型默认并发 `3`：由 `REPORT_LLM_CONCURRENCY` 配置；6 个运营会分两批生成，超过限制的请求在内存队列等待。
- 浏览器抓取完成后立即释放浏览器队列，不等待大模型生成，因此后续运营可以继续读取门店数据。
- 健康检查只进入浏览器队列，并合并同一时刻的重复检查；它不进入模型队列。
- 健康探针在 3 个有效门店间轮换，避免淘宝闪购页面对连续相同搜索值不重复请求造成误报。
- 服务日志只记录门店 ID、队列等待时间、执行时长及活跃/等待数量，不记录 Cookie、API Key 或请求头。

## 7. 运营页面

页面标题：`淘宝闪购运营日报`

首屏包含：

- 门店 ID 输入框
- `查询并生成日报`按钮
- 当前登录状态和服务端健康状态
- 店铺名称、运营、微信群名
- 昨日数据展示区
- 昨日日报展示区
- `复制微信群名`按钮
- `复制日报`按钮

复制按钮必须使用浏览器 Clipboard API，并在成功后显示“已复制”。复制内容要求：

- 复制微信群名：只复制微信群名字文本。
- 复制日报：复制完整日报文本，不截断、不包含 HTML 标签。

页面不显示 Cookie、API Key、SSH 配置或服务器内部路径。接口错误要显示门店 ID、错误原因和“重试”按钮。

## 8. 云端部署流程

1. 通过本机 `CLOUD_SSH_KEY_PATH` 连接云服务器。
2. 创建 `/home/devbox/taobao-shangou-daily-report` 目录。
3. 上传独立目录中的代码、静态页面、`shops.json`、`package.json` 和 `.env.example`。
4. 单独上传应用 `.env`，写入 Cookie 和大模型配置，执行 `chmod 600 .env`。
5. 安装 Node.js 依赖和 Chrome/Chromium、Xvfb。
6. 启动有头 Chrome，CDP 只监听 `127.0.0.1:9222`。
7. 启动 Node 服务监听 `0.0.0.0:8792`。
8. 通过公网地址检查静态页面和 `/api/health`。
9. 用真实门店 ID 完成一次昨日数据查询和日报生成。
10. 检查复制微信群名和复制日报两个按钮。

SSH 私钥只用于第 1 步，不上传到第 2 步及之后的远程目录。

### 8.1 Cookie 与真实门店验收顺序

部署时必须严格按以下顺序执行，任一步失败都停止对外使用：

1. 确认云端 `.env` 存在且权限为 `600`，只检查变量是否存在，不输出变量值。
2. 重启 Chrome 和 Node 服务，确认 Cookie 能从云端配置恢复。
3. 调用 `/api/health`，确认 `chrome`、`loggedIn`、`storeAnalysisPage` 均为 `true`。
4. 使用 `shops.json` 中的真实门店 ID 调用 `/api/report`，确认能读取昨日数据。
5. 校验返回的门店 ID、店铺名、昨日日期、原始字段和日报内容。
6. 再次重启服务，重复第 3、4 步，确认不是只在首次部署时有效。
7. 只有两次真实查询均通过，才允许把公网地址交给运营使用。

## 9. 验收标准

阶段二只有全部满足以下条件才算完成：

1. 公网首页可以打开。
2. `/api/health` 确认 Chrome 在线且淘宝闪购服务商 Cookie 登录有效。
3. 输入 `1330475849` 能返回正确店铺名称、运营、微信群名。
4. 能读取该门店昨日数据，日期、门店 ID、店铺名均正确。
5. 能调用大模型生成淘宝闪购店铺日报。
6. 输入另一个有效门店 ID 时结果不会串店。
7. 复制微信群名按钮只复制群名。
8. 复制日报按钮复制完整日报。
9. API Key、Cookie 和 SSH 私钥不出现在页面、响应、日志或 Git 中。
10. Cookie 失效时健康检查明确失败，修复登录态后可以恢复。
