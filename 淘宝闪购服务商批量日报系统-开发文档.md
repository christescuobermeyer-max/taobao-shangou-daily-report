# 淘宝闪购服务商 · 批量昨日日报系统开发文档

> 目标读者：Codex / 后续开发者。本文档给出足够详细的规格，可据此直接生成批量脚本，或进一步做成"输入门店 ID 即生成昨日日报"的运营自助页面。
> 所有技术事实基于真实抓包与实测验证（服务商单元 `unitId=1132809`，测试门店 `1319751638` 小输胜麺屋，实测日期 2026-07-17）。
> 配套文档：`淘宝闪购服务商门店分析接口接入文档.md`（接口逆向细节）。
> 本目录为可整体迁移的阶段一脚本包，代码、公共库、测试、文档和环境变量模板均位于同一目录；源 Excel 是外部输入文件，不会被修改。
>
> **平台归属说明**：本文档面向淘宝闪购。淘宝闪购后台当前仍使用 `open.shop.ele.me`、`.ele.me` Cookie、`ELEME_ISV_COOKIE` 和 `X-Eleme-*` 等历史技术标识；这些是实际接口的域名、请求头和兼容变量名，不能改成虚构的淘宝域名或请求头。

## 0. 文档用途

本文档描述一个系统，能力是：

**输入一批淘宝闪购门店 ID → 自动获取每个门店昨日运营数据 → 调用大模型生成可发微信群的运营日报 → 每店保存一份日报文本。**

分两个阶段落地：

- **阶段一（本文档主体）**：命令行批量脚本 `isv-daily-report.mjs`。Codex 可据第 4 节直接实现。
- **阶段二（第 7 节）**：Web 自助页面，运营在浏览器输入门店 ID 即可拿到日报。给出架构与接口设计。

## 1. 系统目标与关键设计决策

| 项 | 决策 | 理由 |
|----|------|------|
| 数据源 | 淘宝闪购服务商列表接口 `pageQueryOpenOfflineDetails`，按"昨日"（`dateType=day`） | 实测该接口每行直接返回门店昨日全部指标，**无需逐店免登进店**，最快最稳，且规避免登 token 时效问题 |
| 获取方式 | 驱动已登录 Chrome 的 UI（搜索框输入 ID + 回车），CDP 监听接口响应 | 复用页面运行时生成的 `secretkey`/`timestamp`/`bx_et`/`token`，不逆向动态参数 |
| 登录态 | 复用 `.env` 里 `ELEME_ISV_COOKIE`（淘宝闪购后台 Cookie，变量名沿用技术历史命名），或已手动登录的调试 Chrome | 已验证可行，见接入文档第 2 节 |
| 日报生成 | OpenAI 兼容 API（vectorengine.ai / gpt-5.6-luna） | 已实测跑通，见第 6 节 |
| 日报输出 | 每店一个 `.txt`，命名 `日报-{店铺名}-{昨日yyyyMMdd}.txt` | 已与用户确认 |
| 门店 ID 输入 | 文本文件（一行一个 ID）为主，命令行参数为辅 | 批量友好 |
| 失败处理 | 单店失败（搜不到/接口异常/API 失败）跳过并记入失败清单，不中断整批 | 一个坏 ID 不应拖垮整批 |
| 日报长度 | 保持当前 prompt 输出（完整版，约 2000+ 字） | 用户已确认内容 OK |

## 2. 已验证的技术事实（地基）

以下均为实测确认，实现时可直接依赖：

本节中的 `ele.me`、`lsycm.alibaba.com`、`app-api.shop.ele.me`、`X-Eleme-*` 等名称是淘宝闪购后台实际使用的基础设施标识。它们不表示本系统改为接入其他平台；请求 URL、Cookie 域和请求头必须按抓包结果保留。

### 2.1 数据源接口

```http
POST https://lsycm.alibaba.com/api/oShopAnalysis/open/general/pageQueryOpenOfflineDetails?bx_et=<运行时>
token: <服务商登录态运行时值>
Content-Type: application/json
Referer: https://open.shop.ele.me/
```

请求体（搜索单店 + 昨日维度）关键字段：

```json
{
  "searchWord": "1319751638",
  "current": 1,
  "pageSize": 10,
  "dateType": "day",
  "dateTypeLabel": "昨日",
  "beginDate": "<昨日yyyyMMdd>",
  "endDate": "<昨日yyyyMMdd>",
  "date": "<昨日yyyyMMdd>",
  "orderBy": "gross_amt",
  "orderType": "desc",
  "unitId": "1132809",
  "openUserId": "100100161380818",
  "accountType": "member",
  "source": "PC",
  "secretkey": "<运行时>",
  "uniquekey": "<运行时>",
  "timestamp": "<运行时>"
}
```

**注意**：`secretkey`/`uniquekey`/`timestamp`/`bx_et`/`token` 全是页面运行时生成，**不能硬编码或重放**。这是"必须驱动真实页面"而非"后端裸 HTTP"的根本原因。

### 2.2 响应结构

```json
{ "code": 0, "data": [ { /* 一家门店的昨日指标, 约149字段 */ } ], "totalCount": 1 }
```

`code=0` 成功；`data` 为空数组表示该 `searchWord` 无匹配门店（不在该服务商名下，或 ID 错误）。

### 2.3 昨日指标字段映射（实测值来自小输胜麺屋 2026-07-16）

| 字段 | 含义 | 实测值 | 处理 |
|------|------|--------|------|
| `shop_id` | 门店 ID | 1319751638 | 直接用 |
| `shop_name` | 门店名 | 小输胜麺屋 | 直接用 |
| `city_name` | 城市 | 金华 | 字段名兜底 `cityName`/`city` |
| `gross_amt` | 营业额(元) | 1192.45 | 直接用 |
| `income_amt` | 收入(元) | 482.13 | 直接用 |
| `exp_uv` | 曝光人数 | 1684 | 直接用 |
| `clk_uv` | 进店人数 | 130 | 直接用 |
| `order_user_cnt` | 下单人数 | 25 | 直接用 |
| `order_user_cnt_new` | 新客下单 | 14 | 直接用 |
| `order_user_cnt_old` | 老客下单 | 11 | 直接用 |
| `valid_order_cnt` | 有效订单数 | 28 | 直接用 |
| `clk_exp_rate` | 进店转化率 | 0.077 | 小数，展示乘 100 → 7.7% |
| `ord_clk_rate` | 下单转化率 | 0.192 | 小数，展示乘 100 → 19.2% |

> 埋点信标 `GET https://fourier.taobao.com/ts?url=...` 会先于真实 POST 出现，实现时忽略，只认 `lsycm.alibaba.com/.../pageQueryOpenOfflineDetails` 的 POST 响应。

## 3. 运行前置条件

1. **Node.js ≥ 18**（用到内置 `node:https`、ESM）。依赖 `ws`（仓库已装）。
2. **一个已登录淘宝闪购服务商账号的 Chrome，开着调试端口**：
   ```bash
   chrome.exe --remote-debugging-port=9222 \
     --user-data-dir="<独立profile路径>" \
     https://open.shop.ele.me/manager/base/store-analysis
   ```
   首次需人工登录；之后可用 `eleme-isv-cookie.json` 注入免登（cookie 未过期时）。
3. **门店 ID 清单文件**，如 `shop-ids.txt`：
   ```
   1319751638
   1328600021
   # 以 # 开头的行忽略
   ```
4. **日报大模型 API 配置**通过环境变量传入（不要写死在脚本）：
   ```
   ELEME_ISV_COOKIE=cna=...; nrksid=...; yqksid=...; tfstk=...; isg=...
   VECTORENGINE_API_KEY=sk-...
   VECTORENGINE_BASE_URL=https://api.vectorengine.ai
   VECTORENGINE_MODEL=gpt-5.6-luna
   ```

   `ELEME_ISV_COOKIE` 是淘宝闪购服务商登录态的现有兼容变量名；`VECTORENGINE_API_KEY` 必须填写真实密钥，另外两个变量可使用上面的默认值。

## 4. 批量脚本规格 `isv-daily-report.mjs`

### 4.1 命令行接口

```bash
# 测试王清月前两家门店
node isv-daily-report.mjs --source "E:\\christescuobermeyer\\taobaoshangou-shengxiaozhongdianpu\\淘宝闪购已建群_按运营分类.xlsx" --sheet "王清月" --operator "王清月" --limit 2 --out "E:\\christescuobermeyer\\taobaoshangou-shengxiaozhongdianpu\\淘宝闪购王清月日报测试.xlsx"

# 可选参数
  --source <xlsx>         # 已建群工作簿，默认使用配置的淘宝闪购工作簿
  --sheet 王清月           # 工作表名称，默认王清月
  --operator 王清月        # 运营名称，默认王清月
  --limit 2               # 限制本次处理数量；两家测试必须显式指定 2
  --port 9222              # Chrome 调试端口，默认 9222
  --out <xlsx>             # 输出工作簿，默认源文件同目录的测试文件名
  --date 20260716          # 覆盖昨日日期（默认自动取昨日）
  --delay 2500             # 每店间隔毫秒，默认 2500（避免风控/节流）
  --api-retries 2          # 取数/API 失败后的重试次数，默认 2
  --api-retry-delay 3000   # 每次 API 重试间隔毫秒，默认 3000
  --dry-run                # 只取数据不调 API，仍写入昨日数据列
```

### 4.2 主流程（伪代码）

```
1. 解析参数，读取源工作簿指定工作表，按运营筛选门店并应用 limit
2. 读环境变量拿 API 配置；缺失则报错退出
3. ymd = args.date || yesterdayYmd()
4. 连接 Chrome CDP：
   - GET http://127.0.0.1:<port>/json 找到淘宝闪购后台 open.shop.ele.me 的 page target
   - 若当前不在 store-analysis 页，Page.navigate 过去并等待加载
   - 校验登录态：页面能找到"搜索门店名称/ID"输入框；找不到 → 提示需登录，退出
   - 点击"昨日"时间按钮一次（后续搜索都沿用昨日维度）
5. 对每个 shopId 顺序执行（不并发，避免同页竞态）：
   a. 启用 Network 监听，准备捕获 pageQueryOpenOfflineDetails 响应
   b. 在搜索框写入 shopId（用原生 setter + input 事件），派发 Enter
   c. 等待对应响应（超时 15s）；解析 data[]
   d. data 为空 → 记入失败清单（原因: 未找到门店），continue
   e. row = data[0]；metrics = extractShopMetrics(row, ymd)
   f. dataText = buildDataText(metrics)
   g. 若 --dry-run：打印 metrics，continue
   h. report = await generateReport({...API配置, dataText})
      - API 失败 → 记入失败清单（原因: 生成失败），continue
   i. 写文件 <out>/日报-<shopName>-<ymd>.txt
   j. 记入成功清单
   k. sleep(args.delay)
6. 打印汇总：成功 N 家、失败 M 家；失败清单（shopId + 原因）
7. 断开 CDP，退出码：全成功=0，有失败=1
```

### 4.3 CDP 捕获响应的实现要点

用**单个页面级 WebSocket 连接**同时驱动 UI 和监听网络：

```
- 连接 page target 的 webSocketDebuggerUrl
- send("Network.enable") + send("Runtime.enable") + send("Page.enable")
- 监听 Network.responseReceived：记录 url 含 pageQueryOpenOfflineDetails 的 requestId
- 监听 Network.loadingFinished：对匹配的 requestId 调 Network.getResponseBody 取 body
- 用 Promise + requestId 关联：每次搜索前生成一个 pending resolver，
  loadingFinished 命中目标 url 时 resolve(body)
- 注意：一次搜索可能先有 fourier.taobao.com/ts 埋点，过滤掉，只认 lsycm 域的 POST
```

搜索框写值（React 受控组件，必须用原生 setter 触发）：

```js
const inp = [...document.querySelectorAll('input')]
  .find(e => e.placeholder === '搜索门店名称/ID');
const setter = Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value').set;
setter.call(inp, shopId);
inp.dispatchEvent(new Event('input', { bubbles: true }));
inp.dispatchEvent(new KeyboardEvent('keydown', { key: 'Enter', keyCode: 13, bubbles: true }));
```

点"昨日"按钮：

```js
[...document.querySelectorAll('span,button,a,div')]
  .find(e => e.children.length === 0 && e.innerText?.trim() === '昨日')?.click();
```

### 4.4 边界与健壮性

| 情况 | 处理 |
|------|------|
| Chrome 未开调试端口 | 启动即检测 `/json/version`，失败给出明确提示（如何启动） |
| 未登录（找不到搜索框） | 明确提示"请先在 Chrome 登录服务商账号"，退出 |
| 门店 ID 搜不到（data 空） | 记失败清单，继续 |
| 接口超时（15s 无响应） | 记失败清单，继续；可选重试 1 次 |
| 一个 ID 搜索后残留在框里 | 每次搜索前先清空输入框 |
| API 限流/超时 | 取数接口和大模型 API 默认各重试 2 次，仍失败则记入失败清单并继续 |
| 店铺名含非法文件名字符 | 文件名做 sanitize（替换 `/ \ : * ? " < > |`） |
| 同名门店 | 文件名带上 shopId 去重：`日报-{name}-{shopId}-{ymd}.txt` |

## 5. 数据处理规格（已在 isv-report-lib.mjs 实现）

`extractShopMetrics(row, ymd)` 输出的标准指标对象：

```js
{
  date, city, shopName, shopId,
  grossAmt, incomeAmt, expUv, clkUv,
  orderUserCnt, validOrderCnt,
  clkExpRate,   // 已格式化 "7.7%"
  ordClkRate,   // 已格式化 "19.2%"
  orderUserNew, orderUserOld
}
```

`buildDataText(metrics)` 输出给大模型的文本（示例）：

```
日期：20260716
城市：金华
店铺名称：小输胜麺屋
店铺ID：1319751638
营业额：1192.45 元
曝光人数：1684 人
入店人数：130 人
入店转化率：7.7%
下单人数：25 人（新客 14 / 老客 11）
下单转化率：19.2%
有效订单数：28 单
```

字段缺失时以 `-` 兜底，不要让 `undefined` 进日报。

阶段一输出的 `昨日数据` 列不是只保存上述摘要，而是每家门店一个单元格，保存昨日日期、门店 ID、接口返回店铺名，以及接口 `data[0]` 全部字段对应的中文数据名称和原始值。大模型输入同时包含完整中文字段数据和上述标准指标摘要，不再直接使用英文字段名。

## 6. 大模型 API 规格（已实测）

| 项 | 值 |
|----|----|
| Base URL | 环境变量 `VECTORENGINE_BASE_URL`，默认 `https://api.vectorengine.ai` |
| 路径 | `/v1/chat/completions`（OpenAI 兼容） |
| Model | 环境变量 `VECTORENGINE_MODEL`，默认 `gpt-5.6-luna` |
| 认证 | `Authorization: Bearer <VECTORENGINE_API_KEY>` |
| Key | **必须通过环境变量 `VECTORENGINE_API_KEY` 传入，不写死**；真实密钥不得写入文档或提交仓库 |
| 实测延迟 | 单次约 30-35s |
| 输出 | 纯文本日报，约 2000+ 字 |

请求体：

```json
{
  "model": "gpt-5.6-luna",
  "messages": [
    { "role": "system", "content": "<SYSTEM_PROMPT>" },
    { "role": "user", "content": "请根据以下运营数据生成日报：\n\n<dataText>" }
  ],
  "temperature": 0.7
}
```

`SYSTEM_PROMPT` 已存于 `isv-report-lib.mjs`（用户提供的完整运营分析助手 prompt）。响应取 `choices[0].message.content`。

### 6.1 API 配置审计结论

日报生成所需的可变配置统一放在环境变量中：

- `VECTORENGINE_API_KEY`：必填，当前 `.env` 已配置非空值；密钥内容不写入文档，是否有效仍需在实际调用时验证。
- `VECTORENGINE_BASE_URL`：已纳入环境变量，默认值为 `https://api.vectorengine.ai`。
- `VECTORENGINE_MODEL`：已纳入环境变量，默认值为 `gpt-5.6-luna`。
- `ELEME_ISV_COOKIE`：淘宝闪购服务商登录态，已存在于当前 `.env`；该变量名保留是为了兼容真实 Cookie 域和现有资产。

以下内容不应放进环境变量：淘宝闪购后台固定接口 URL、OpenAI 兼容路径 `/v1/chat/completions`、以及页面运行时生成的 `secretkey`、`uniquekey`、`timestamp`、`bx_et` 和 `token`。前两者是固定协议配置，后者必须由真实页面运行时生成，不能硬编码或重放。

当前 `isv-report-lib.mjs` 的 `generateReport()` 通过参数接收 `baseUrl`、`apiKey` 和 `model`，尚未自行读取 `.env`；阶段一主脚本必须从 `process.env.VECTORENGINE_*` 读取并校验后再传入。

> 因单次 30s+，批量 N 家串行耗时 ≈ N × (30s + delay)。50 家约 30 分钟。若要提速，可对**API 调用**做有限并发（如 3-5 并发），但**页面搜索取数必须串行**（同一页面同一输入框）。推荐结构：先串行取完所有门店数据，再并发生成日报。

## 7. 阶段二：运营自助 Web 系统设计

目标：运营在浏览器输入门店 ID（单个或批量）→ 点击生成 → 拿到日报（可复制/下载/一键发群）。

### 7.1 架构

```
运营浏览器 (前端页面)
   │ HTTP: POST /api/report {shopIds:[...]}
   ▼
Node 后端服务 (Express/Fastify)
   │ ① 复用 isv-report-lib.mjs
   │ ② 持有一个常驻的已登录 Chrome (调试端口)
   │ ③ 串行取数 + 并发生成日报
   ▼
已登录淘宝闪购服务商 Chrome (CDP) ──▶ open.shop.ele.me 取昨日数据
大模型 API ──▶ 生成日报
   │
   ▼ 返回 [{shopId, shopName, report, status}]
前端展示 / 下载 / 复制
```

### 7.2 后端接口设计

```
POST /api/report
  body: { "shopIds": ["1319751638", ...], "date": "20260716"(可选) }
  resp: { "results": [
           { "shopId","shopName","report","status":"ok" },
           { "shopId","status":"not_found" },
           { "shopId","status":"error","message":"..." }
         ] }

GET /api/health
  检查 Chrome 登录态是否有效（能否找到搜索框），返回 {loggedIn:bool}

POST /api/login-check
  返回当前服务商单元信息，供前端显示"当前登录：xxx服务商"
```

支持 **SSE / WebSocket 推送进度**（每完成一家推一条），因为批量耗时长，前端要有进度条。

### 7.4 关键工程约束

| 约束 | 说明 |
|------|------|
| Chrome 单例串行取数 | 一个后端进程持一个 Chrome，搜索取数排队串行，避免输入框竞态 |
| 登录态维护 | 淘宝闪购后台 cookie 会过期（含风控令牌）。需检测失效并提供"重新登录"入口（人工在服务器 Chrome 登录，或注入新 cookie bundle） |
| 密钥管理 | API key、cookie 存服务端环境变量/密钥管理，绝不下发前端 |
| 并发生成 | 取数串行、生成日报可并发（限 3-5），平衡速度与限流 |
| 多服务商 | 若要支持多个服务商账号，需多 Chrome profile + 多端口，或按账号切换 cookie（同一时刻只一个登录态） |
| 审计 | 记录谁在什么时间为哪些门店生成了日报 |

### 7.5 从阶段一到阶段二的演进

阶段一脚本的核心逻辑（CDP 取数 + 日报生成）可**原样抽成后端 service 模块**：
- 取数逻辑 → `fetchShopMetrics(cdpSession, shopId, ymd)`
- 日报逻辑 → 已在 `isv-report-lib.mjs`
- Web 层只是加了 HTTP 入口 + 常驻 Chrome + 进度推送

即：**先把阶段一脚本写好且模块化，阶段二 = 套一层 HTTP 服务 + 前端页面。**

## 8. 风险与注意事项

| 项 | 说明 |
|----|------|
| 动态参数 | `secretkey`/`timestamp`/`bx_et`/`token`/`uniquekey` 运行时生成，禁止硬编码或重放。必须驱动真实页面 |
| 登录态时效 | 服务商 cookie 含风控令牌（`tfstk`/`isg`），会过期，需检测失效并重新登录 |
| 风控节流 | 批量搜索加间隔（默认 2.5s/店），避免触发风控 |
| 页面竞态 | 同一输入框串行搜索，每次先清空再输入，等响应回来再下一个 |
| 密钥安全 | API key、cookie 走环境变量/密钥管理；`.env`/`eleme-isv-cookie.json` 已 gitignore |
| 数据口径 | 指标口径来自"生意参谋"；`order_user_cnt`(下单人数) 与 `valid_order_cnt`(有效订单数) 可能不等（一人多单），属正常 |
| 日期时区 | `yesterdayYmd()` 用本地时区；服务器部署注意时区设置为 Asia/Shanghai |
| 文件名安全 | 店铺名 sanitize 非法字符，同名带 shopId 去重 |

## 9. 验收标准

阶段一脚本视为完成，当且仅当：

1. `node isv-daily-report.mjs --ids 1319751638 --dry-run` 能打印出该店昨日正确指标（营业额 1192.45 等，日期为运行时昨日）。
2. 去掉 `--dry-run` 能在 `./reports/` 生成 `日报-小输胜麺屋-<昨日>.txt`，内容为结构化日报。
3. 传入含"不存在的门店 ID"的清单，该 ID 被记入失败清单，其余门店正常生成，脚本不崩。
4. 传入多个有效 ID，全部各生成一份日报，末尾打印成功/失败汇总。
5. Chrome 未开调试端口 / 未登录时，给出明确可操作的错误提示，不抛裸异常栈。
