# 淘宝闪购日报生成网关开发文档

**版本：** v1.1

**文档状态：** 已实现并部署上线

**适用系统：** 淘宝闪购阶段二云端日报系统

## 1. 项目目标

在现有门店 ID 输入区下方新增“日报生成网关”。运营点击“查询并生成日报”后，可以实时看到当前系统正在处理哪些门店、各门店已经花费的时间、当前有多少任务正在生成或排队、自己的日报处于哪个阶段、自己已经等待多久，以及近期日报的平均生成时间。

该功能只增加任务可见性，不改变现有日报生成结果、浏览器并发 1、大模型并发 3、缓存、重试和 Cookie 登录方式。

## 2. 已确认范围

### 2.1 必须完成

- 网关位于门店 ID 输入框和“查询并生成日报”按钮下方。
- 实时显示所有未结束任务对应的完整门店 ID。
- 每个任务显示当前阶段和从提交开始累计的耗时。
- 显示当前任务总数、正在调用大模型的数量、排队等待数量。
- 当前页面提交的任务必须明显标识为“我的日报”。
- 显示本人已经等待或已经处理的时间。
- 显示近期日报平均生成时间。
- 日报成功或失败后，网关状态与原有结果区、错误提示保持一致。
- 不能影响现有 6 人并发稳定性，不能增加浏览器/CDP 并发。

### 2.2 本版本暂不包含

- 不提供取消、暂停、插队、调整优先级功能。
- 不增加运营账号、登录、角色权限或管理员页面。
- 不提供历史日报任务查询、筛选、导出或图表。
- 不公开店铺名称、运营姓名、微信群名、日报正文、Cookie、API Key 或服务商请求信息。
- 不把同步 `/api/report` 改造成异步任务接口，现有调用方式保持兼容。

## 3. 核心设计结论

### 3.1 产品命名

页面名称使用“日报生成网关”，帮助运营理解它是所有日报任务进入生成服务的统一状态入口。技术实现不是独立反向代理网关，而是服务端任务状态注册表、统计模块和只读状态 API。

### 3.2 实时通信方式

采用短轮询，不采用 WebSocket 或 SSE：

- 本人日报进行中：每 1 秒请求一次状态。
- 当前页面没有本人任务：每 2 秒请求一次状态。
- 浏览器页面隐藏：每 5 秒请求一次状态。
- 每次请求结束后再安排下一次，禁止产生重叠轮询。
- 状态接口失败后保留最后一次成功内容，显示“状态更新中断”，5 秒后继续尝试。

选择短轮询的原因：当前系统只有约 6 名并发运营，状态响应很小；原生 Node HTTP 和现有 Sealos 代理无需新增长连接配置，部署与断线恢复更稳定。

### 3.3 本人任务识别

前端每次提交前使用 `crypto.randomUUID()` 生成 `requestId`，并随日报请求提交：

```json
{
  "shopId": "1330475849",
  "requestId": "5cb5eb64-6b66-4ce7-bbb0-64cc95c8615f"
}
```

`requestId` 只用于当前请求与网关状态关联，不作为登录凭证，不写入日报缓存。服务端必须校验其格式和长度；旧调用方不传 `requestId` 时仍可正常生成日报，但无法在页面中标识“我的日报”。

### 3.4 重复门店处理

执行任务按 `shopId-date` 唯一。若两个页面同时请求同一门店同一天的数据：

- 继续复用现有去重逻辑，只抓取一次、调用一次大模型。
- 网关全局列表只显示一个执行任务，不能把它统计成两家正在生成。
- 两个 `requestId` 都订阅这个执行任务，两个页面都将其识别为“我的日报”。

## 4. 页面布局

网关放在现有搜索行下方、错误提示上方，属于搜索工作区的一部分，不新建页面、不放进结果区。

```text
门店 ID
[ 1330475849                         ] [查询并生成日报]

日报生成网关                                      实时
┌──────────────────────────────────────────────────────┐
│ 当前任务 5          正在生成 3 / 3       近20份平均 43秒 │
│ 排队等待 2                                            │
├──────────────────────────────────────────────────────┤
│ 我的日报  1330475849  等待生成  已等待 00:38  前面 2 个 │
│             预计约 01:10 后完成                       │
├──────────────────────────────────────────────────────┤
│ 1330475849  等待生成   00:38   我的                    │
│ 544882229   正在生成   00:31                           │
│ 544976455   正在生成   00:22                           │

```

### 4.1 信息层级

1. 第一层是三项摘要：当前任务、正在生成、平均生成时间。
2. 第二层是“我的日报”，只在当前页面有进行中或刚结束任务时显示。
3. 第三层是当前任务列表，按提交时间从早到晚排列。

### 4.2 视觉规范

- 网关使用现有搜索面板内的无框分区，通过上边界和 20-24px 间距分组，禁止在面板中再嵌套浮动卡片。
- 摘要数字使用 20px/600，标签使用 12-13px/500，任务列表使用等宽数字字体显示门店 ID 和计时。
- 正在生成使用成功绿，等待使用警示黄，正在读取使用信息蓝，失败使用错误红；品牌橙色只用于“我的”标识和关键焦点。
- 状态不能只靠颜色区分，必须同时显示中文状态文字。
- “我的日报”行使用浅品牌色背景或左侧 3px 强调条，不使用强投影。
- 移动端 360px 下摘要改为两列换行，任务行允许状态和计时换到第二行，门店 ID 不截断。
- 动效只用于 150-200ms 状态切换；遵循 `prefers-reduced-motion`，禁止持续跳动或大面积闪烁。

### 4.3 空状态与完成状态

- 无任务：显示“当前没有正在处理的日报”，平均值仍保留。
- 本人任务成功：显示“日报已生成”，保留 3 秒后收起“我的日报”区，原有结果区正常展示。
- 本人任务失败：显示“生成失败”和可读错误，原有错误提示仍是主要错误出口。
- 缓存命中：显示“已从缓存读取”，不进入排队统计，不影响平均生成时间。

## 5. 任务状态模型

服务端内部状态使用稳定枚举，页面统一转换为中文文案：

| 内部状态 | 页面文案 | 是否计入当前任务 | 计时口径 |
|---|---|---:|---|
| `waiting_browser` | 等待读取 | 是 | 从任务提交开始 |
| `fetching_browser` | 正在读取 | 是 | 从任务提交开始；另记阶段耗时 |
| `waiting_llm` | 等待生成 | 是 | 从任务提交开始；另记模型队列等待 |
| `generating_llm` | 正在生成 | 是 | 从任务提交开始；另记模型生成耗时 |
| `completed` | 已生成 | 否 | 最终总耗时 |
| `failed` | 生成失败 | 否 | 失败前总耗时 |
| `cache_hit` | 已从缓存读取 | 否 | 缓存读取耗时 |

状态只能按以下方向推进，禁止回退：

```text
waiting_browser
  -> fetching_browser
  -> waiting_llm
  -> generating_llm
  -> completed

任一非终态 -> failed
缓存命中 -> cache_hit
```

重试发生在 `generating_llm` 内部，页面状态保持“正在生成”，可将文案更新为“正在重试生成”，但不能把同一个任务重复加入队列。

## 6. 数据模型

### 6.1 执行任务 `GatewayExecution`

| 字段 | 类型 | 说明 |
|---|---|---|
| `key` | string | `shopId-date`，唯一执行键 |
| `shopId` | string | 5-20 位门店 ID |
| `date` | string | 昨日日期 `yyyyMMdd` |
| `status` | enum | 当前状态 |
| `submittedAt` | number | 服务端接收时间戳 |
| `stageStartedAt` | number | 当前阶段开始时间戳 |
| `browserStartedAt` | number/null | 浏览器抓取开始时间 |
| `llmQueuedAt` | number/null | 进入模型队列时间 |
| `llmStartedAt` | number/null | 获得模型并发槽时间 |
| `completedAt` | number/null | 成功或失败时间 |
| `requestIds` | Set<string> | 订阅当前执行任务的页面请求 ID |
| `errorCode` | string/null | 非敏感错误分类，不保存完整堆栈 |

活动任务只保存在进程内存。完成或失败任务保留 5 分钟用于页面展示，之后自动清理；内存中最多保留 100 条终态任务。

### 6.2 耗时指标 `GatewayMetrics`

保存路径：`data/gateway-metrics.json`，文件权限 `600`，使用临时文件加重命名原子写入。

```json
{
  "version": 1,
  "modelDurationsMs": [41712, 38590, 45797],
  "updatedAt": "2026-07-17T10:12:00.000Z"
}
```

仅保留最近 100 个成功模型阶段耗时，不保存门店 ID、请求 ID、日报内容、错误内容或任何凭证。多个模型任务可能同时完成，指标更新必须进入同一条内存写队列串行执行；每次写入使用唯一临时文件再原子重命名，禁止多个任务直接覆盖同一临时文件。

## 7. 统计口径

### 7.1 当前任务数

`waiting_browser`、`fetching_browser`、`waiting_llm`、`generating_llm` 的唯一执行任务总数。

### 7.2 正在生成数

状态为 `generating_llm` 的任务数。页面同时显示模型并发上限，例如 `3 / 3`。

### 7.3 排队等待数

`waiting_browser` 与 `waiting_llm` 的任务总数；正在读取和正在生成不计入等待。

### 7.4 平均生成时间

- 使用最近 20 个成功且非缓存的模型阶段耗时计算算术平均值。
- 从获得模型并发槽开始，到最终成功生成日报结束。
- 模型超时后的重试和重试间隔包含在本次实际耗时内。
- 失败任务和缓存命中不参与平均值。
- 样本少于 3 个时显示“统计中”，同时显示样本进度，例如 `2 / 3`。
- 页面文案固定为“近20份平均生成”，避免被理解为浏览器取数或整个请求的平均时间。

### 7.5 本人等待与预计时间

- `elapsedMs` 始终是服务端当前时间减去任务提交时间，页面显示为 `mm:ss`。
- 处于等待状态时文案为“已等待”，处于读取或生成状态时文案为“已用时”。
- `queuePosition` 是本人在当前阶段等待队列中的位置，从 1 开始；正在执行时为 0。
- 预计时间只在至少有 3 个平均耗时样本后显示，并必须带“约”。
- 本人仍在浏览器等待或读取阶段时，只显示精确的已等待/已用时间，不给出不可靠的预计完成时间。
- 本人正在生成时，预计剩余时间为 `max(平均生成时间 - 本阶段已用时, 0)`。
- 本人等待模型时，服务端先为 3 个模型槽计算预计可用时间：每个正在生成任务取 `max(平均生成时间 - 该任务已生成时间, 0)`，空闲槽取 0；再按 FIFO 顺序把等待任务依次分配给最早可用槽，每分配一个任务就给该槽增加一次平均生成时间。本人的预计剩余时间等于模拟后分配到的预计完成时间。
- 估计值小于 0 时按 0 处理；模型耗时波动较大，页面不得显示精确完成时刻或承诺性文案。

## 8. API 设计

### 8.1 修改 `POST /api/report`

新增可选字段 `requestId`，不改变原成功响应和错误响应：

```json
{
  "shopId": "1330475849",
  "requestId": "5cb5eb64-6b66-4ce7-bbb0-64cc95c8615f"
}
```

校验规则：

- `shopId` 保持现有 5-20 位数字规则。
- `requestId` 可选；传入时必须是 UUID，或限制为 8-64 位字母、数字、短横线和下划线。
- 非法 `requestId` 返回 HTTP 400，不创建任务。
- 缓存命中直接返回现有日报结果，不进入活动队列。

### 8.2 新增 `GET /api/gateway`

请求：

```text
GET /api/gateway?requestId=5cb5eb64-6b66-4ce7-bbb0-64cc95c8615f
```

响应：

```json
{
  "ok": true,
  "serverTime": "2026-07-17T10:10:30.000Z",
  "summary": {
    "current": 5,
    "generating": 3,
    "waiting": 2,
    "llmConcurrency": 3,
    "averageGenerationMs": 43210,
    "averageSampleSize": 20
  },
  "me": {
    "shopId": "1330475849",
    "status": "waiting_llm",
    "statusText": "等待生成",
    "elapsedMs": 38000,
    "stageElapsedMs": 32000,
    "queuePosition": 2,
    "estimatedRemainingMs": 70100
  },
  "jobs": [
    {
      "shopId": "1330475849",
      "status": "waiting_llm",
      "statusText": "等待生成",
      "elapsedMs": 38000,
      "isMine": true
    },
    {
      "shopId": "544882229",
      "status": "generating_llm",
      "statusText": "正在生成",
      "elapsedMs": 31000,
      "isMine": false
    }
  ]
}
```

接口要求：

- 只读、无副作用，响应头使用 `Cache-Control: no-store`。
- 不调用 Chrome、不调用大模型、不进入任何并发队列。
- `jobs` 只返回非终态唯一执行任务，按 `submittedAt` 升序，最多 20 条。
- 超过 20 条时 `summary.current` 保持真实总数，响应增加 `hiddenCount`。
- 不返回内部执行键、其他用户的 `requestId`、店名、运营、群名、日报或错误堆栈。
- 目标响应时间：云端 P95 小于 200ms。

## 9. 服务端实现方案

新增模块：

```text
cloud/report-gateway.mjs
cloud/report-gateway.test.mjs
data/gateway-metrics.json       运行后生成，不提交 Git
```

`report-gateway.mjs` 负责：

- 创建任务、绑定 `requestId`、合并同键任务。
- 校验状态只能向前流转。
- 计算当前列表、队列位置、个人耗时和摘要。
- 记录最近 100 次成功模型耗时并原子落盘。
- 清理超过 5 分钟的终态任务。
- 生成不包含敏感字段的 API 结果。

在现有 `cloud/server.mjs` 中增加以下埋点，不解析日志：

1. 请求校验通过且缓存未命中：注册 `waiting_browser`。
2. 浏览器限流任务真正开始：切换 `fetching_browser`。
3. 原始数据获取成功、准备进入模型限流器：切换 `waiting_llm`。
4. 模型限流任务真正开始：切换 `generating_llm`，记录 `llmStartedAt`。
5. 日报生成并写入缓存成功：切换 `completed`，记录模型耗时样本。
6. 任一阶段抛错：切换 `failed`，保存错误分类后继续使用现有 502 响应。
7. 新增 `/api/gateway` 路由读取快照。

现有浏览器限流器、大模型限流器、日报去重器和健康检查逻辑保持不变。

## 10. 前端实现方案

修改 `cloud/public/index.html`：

1. 在搜索行下方增加语义化 `<section id="gateway">`。
2. 使用 `crypto.randomUUID()` 创建本次 `requestId`。
3. 发起 `/api/report` 前立即显示“正在提交”，并启动网关轮询。
4. 每次状态响应只使用 `textContent` 更新文本，禁止使用未清洗的 `innerHTML`。
5. 本人任务状态发生阶段变化时，通过独立 `aria-live="polite"` 区域播报一次；每秒计时变化不触发屏幕阅读器播报。
6. 报告请求结束后继续刷新一次，显示完成或失败状态，再恢复空闲轮询频率。
7. 使用递归计时安排下一次轮询，页面卸载时停止；网络慢时不产生重复请求。
8. 原查询按钮的 loading、disabled 和错误处理保持不变。

## 11. 异常与边界处理

| 场景 | 处理方式 |
|---|---|
| 状态接口临时失败 | 保留最后快照，显示“状态更新中断”，5 秒后重试，不中断日报 POST |
| 日报接口失败 | 本人状态切换失败，原错误区显示具体原因，任务从活动列表移除 |
| 页面刷新或关闭 | 后端任务继续执行并正常缓存；新页面可以看到全局门店 ID，但不会自动识别旧请求为“我的” |
| PM2 重启 | 内存活动任务清空；进行中的 HTTP 请求失败，前端按现有错误流程处理；平均耗时从指标文件恢复 |
| Cookie 失效 | 浏览器阶段失败，不进入模型队列；网关显示失败，健康检查继续返回 503 |
| 模型重试 | 状态保持正在生成，耗时继续累计，不新增任务行 |
| 相同门店重复提交 | 全局只显示一个执行任务，每个提交页面都可识别为自己的任务 |
| 缓存命中 | 快速返回结果，不计入当前任务和平均耗时 |
| 平均样本不足 | 显示“统计中”，不显示预计剩余时间 |

## 12. 安全与性能要求

- 网关 API 不返回 Cookie、API Key、token、请求头、店名、运营、群名或日报内容。
- 服务日志可以记录状态流转、门店 ID 和耗时，不记录 `requestId` 全值；需要关联时只记录前 8 位。
- `requestId` 必须严格校验，不能参与文件路径、命令或动态代码执行。
- 指标文件只保存数字耗时和更新时间，权限为 `600`。
- 轮询接口只读取内存和小型统计数据，不进行同步大文件读取。
- 活动任务与终态任务都有数量和时间上限，防止长期运行导致内存增长。
- 页面只用 `textContent` 渲染门店 ID 和状态，维持现有 CSP 和同源访问。
- 当前系统暂不增加访问口令，网关遵循现有公司内部使用边界。

## 13. 测试方案

### 13.1 单元测试

- 状态按合法顺序推进，非法回退被拒绝。
- 同一 `shopId-date` 合并为一个执行任务。
- 多个 `requestId` 能映射到同一任务并各自得到 `isMine=true`。
- 当前任务、正在生成、等待数量计算正确。
- 队列位置按提交顺序计算正确。
- 最近 20 次平均值、少于 3 次“统计中”、最近 100 次持久化正确。
- 失败与缓存命中不进入平均值。
- 终态 5 分钟清理和最多 100 条限制正确。
- 网关响应不包含敏感字段。

### 13.2 API 测试

- 不传 `requestId` 的旧 `/api/report` 请求仍成功。
- 合法 `requestId` 能在 `/api/gateway` 找到本人任务。
- 非法 `requestId` 返回 400。

- 无任务、1 个任务、6 个任务和超过 20 个任务时响应结构正确。

## 15. 验收标准

- 运营点击生成后 1 秒内能在网关看到自己的门店 ID。
- 网关能准确区分等待读取、正在读取、等待生成和正在生成。
- 当前任务、正在生成、排队等待三个数量与服务端实际状态一致。
- 自己的已等待/已用时间误差不超过 2 秒。
- 至少 3 个成功样本后显示平均生成时间，计算结果与最近 20 次样本一致。
- 同门店重复请求不重复计数、不重复调用模型。
- 6 个不同门店并发时浏览器并发仍为 1、模型并发仍不超过 3。
- 网关故障不会中断日报生成，日报失败也不会让轮询无限运行。
- 页面和网关响应不泄露凭证、日报正文或非必要门店资料。
- PM2 重启后平均耗时能够从指标文件恢复。
